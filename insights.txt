1. PHI-3 EXTRACTIVE COMPRESSOR — VERIFICATION FAILURE
--------------------------------------------------------------------------------

PROBLEM:
  phi3:3.8b failed extractive verification 100% of the time at 5% threshold.
  The model routinely inserts function words ("the", "is", "and", connectors)
  even when explicitly told to output only verbatim source tokens. This is a
  fundamental limitation of small LLMs doing instruction-following for
  extractive tasks.

ROOT CAUSE (two-fold):
  a) 5% max_violation_rate was too strict. Even a few inserted function words
     push the novel-token fraction above 5%.
  b) No post-processing existed to clean up minor insertions. Outputs were
     either accepted as-is or rejected entirely — no middle ground.

FIX (commit 136305f):
  - Relaxed max_violation_rate from 5% to 15% (gate, not guarantee)
  - Added _strip_novel_tokens() to mechanically remove non-source words
  - Pipeline: model generates -> relaxed gate filters -> stripping enforces
    100% extractiveness on the final output
  - Net effect: outputs that are 85%+ extractive pass gate, then get cleaned
    to be strictly extractive

THESIS NARRATIVE VALUE:
  - Report the raw verification failure rate as evidence that small LLMs
    cannot reliably follow extractive constraints
  - Log fraction of tokens stripped per ratio as a mini-finding
  - This supports the thesis argument that compression is non-trivial even
    with explicit constraints

RISK:
  After stripping, Phi-3 extractive becomes "LLM-guided span selector +
  mechanical filter" — potentially similar to LLMLingua-2 (also a token-level
  selector). If both produce statistically indistinguishable outputs across
  all ratios, you effectively have only 2 distinct compressors, weakening H2.
  Monitor this in the full sweep.

2. MODEL CHOICE — KEEP PHI-3, DON'T SWITCH
--------------------------------------------------------------------------------

DECISION: Keep Phi-3-Mini-3.8B for both compressor and H5 mid-size planner.

REASONING:
  - Thesis novelty is in the experimental framework (coordination cliff,
    compounding-error model, inference disclosure), not model choice
  - Reviewers won't care whether it's Phi-3 vs Gemma-2 vs Qwen-2.5
  - Phi-3 at 3.8B is a natural midpoint in the 1.5B/3.8B/8B H5 ladder
  - MIT license, clean for academic use
  - Switching costs time with minimal benefit on a 4-week timeline
  - The verification struggles are themselves interesting data

ALTERNATIVES CONSIDERED:
  - Qwen2.5-3B: potentially better instruction-following, but smaller gap
    from 1.5B in H5 ladder
  - Gemma-2-2B: same concern, plus different tokenizer family
  - Phi-3.5-Mini: marginal improvement, not worth re-validation effort

### H5 — MODEL-SIZE SCALING (CLEAR SIGNAL)

  Config: 8B planner only (smoke), lingua2 compressor, 3 workloads, 4 ratios
  Results:
    ratio=1.0:  coord_success=1.0, F1=0.997-0.999 (near perfect)
    ratio=4.0:  coord_success=0.0, F1=0.01-0.27  (cliff!)
    ratio=8.0:  coord_success=0.0, F1=0.0         (total failure)
    ratio=16.0: coord_success=0.0, F1=0.0-0.65    (hallucinated numbers)

  INSIGHT: The coordination cliff is REAL and very sharp — between 1x and 4x.
  The 8B planner correctly aggregates hours/budget from uncompressed fragments
  (F1~1.0) but fails completely at 4x compression because LLMLingua-2 drops
  the numeric tokens that are essential for the aggregation task.

  The non-zero F1 at 16x is noise — the model hallucinates plausible numbers
  that partially match by coincidence.

  VERDICT: H5 infrastructure works end-to-end. Ready for full 3-model sweep.

### H1/H2 — COORDINATION CLIFF SWEEP (PARTIAL SIGNAL)

  Config: 3 compressors (lingua2, phi3-extractive, filter), 2 workloads,
          3 ratios (1x, 4x, 8x), 1 seed, family-a, deterministic solver

  Results:
    All compressors: coord_success=1.0 at ratio=1x, 0.0 at ratio>=4x
    qa_f1 = 0.0 everywhere (BUG — see section 12)
    token_recall = 0.0 everywhere (BUG — see section 12)
    subtask_acc = 1.0 everywhere (deterministic solver cheats)

  INSIGHT: The deterministic solver correctly demonstrates the cliff effect.
  All 3 compressors show identical cliff behavior because the deterministic
  solver uses regex to extract "hours: X" and "budget: EUR Y" from compressed
  text — when compression removes these patterns, it scores 0.

  Phi3-extractive took ~3 minutes for 16 fragments (2 workloads x 8 frags)
  via Ollama. At full scale (150 workloads x 8 frags x 10 ratios), that's
  ~4.5 hours just for phi3 compression, plus lingua2 and filter time.

  H1 shows rho=0.0 (constant arrays) — not meaningful with only 2 workloads.
  H2 shows no cliff detected (tau=N/A) — piecewise fit needs >=4 ratio points.

  VERDICT: Deterministic solver works. Need more ratios and workloads for
  statistical tests to fire. The qa_f1 metric is broken (see section 12).

### H3 — RAG PIPELINE PLACEMENT (BROKEN METRIC)

  Config: lingua2 + filter, 5 workloads, storage/accuracy regimes
  Results: F1 = 0.0 across ALL pipelines, compressors, regimes

  ROOT CAUSE: H3 computes F1(retrieved_text, expected_answer). The expected
  answer is "hours=819;budget=170969" (aggregated). The retrieved text is
  raw fragments like "Recorded hours: 174. Approved budget: EUR 5608."
  These share zero tokens — F1 is always 0.

  The retrieval pipeline WORKS — it finds and returns relevant fragments.
  The bug is in the scoring, not the retrieval.

  FIX NEEDED: Either:
    a) Change F1 to measure retrieval recall (did we retrieve the right
       fragments?) rather than answer quality
    b) Add an LLM reader after retrieval to aggregate the answer, then
       score that against expected_answer
    c) Use fragment-level F1: for each retrieved fragment, compute F1
       against the original source fragment to measure compression quality

  VERDICT: Pipeline works, metric is wrong. Must fix before full run.

### H4 — INFERENCE DISCLOSURE (QUESTION DESIGN PROBLEM)

  Config: lingua2 only, 3 workloads, 4x compression, llama3.1:8b reader
  Results:
    priors_correct = 1.00 (18/18 correct with NO fragment text)
    baseline_correct = 0.94 (17/18 with full text)
    compressed_correct = 0.89 (16/18 with compressed text)

  ROOT CAUSE: ALL ground-truth answers are "no". The reader model defaults
  to "no" when it has insufficient information, so priors get 100%. The
  metric can't distinguish "I don't know" from "I verified it's no."

  Example questions:
    "Did contract_db for project P-1794 exceed EUR 27204 in budget?" → no
    "Was contract_db's recorded hours for P-1794 more than 65?" → no

  FIX NEEDED: The C1 benchmark's protected_facts generator must produce
  a balanced mix of "yes" and "no" questions. Currently it generates only
  "Did X exceed Y?" questions with thresholds above the actual values.

  ALSO NOTABLE: Compression IS working — compressed_text_len = 18-32 chars
  vs original = 141-162 chars (actual 4-8x compression ratio).

  VERDICT: Ollama reader works, compression works, but question design
  makes the metric vacuous. Must fix question generation.

5. H2 CLIFF DETECTION — CONFIRMED REAL
--------------------------------------------------------------------------------

From micro experiments (2026-05-23):
  - H5 with real Ollama planner: coord_success drops from 1.0 to 0.0
    between ratio 1x and 4x. Sharp cliff, no gradual decline.
  - H1/H2 with deterministic solver: same pattern, all 3 compressors.
  - The cliff is at ~2x-4x for family-a (numeric aggregation tasks).

From the smoke test (h1_h2_smoke, May 21):
  - lingua2 shows tau* ~ 16.0 (extreme end of ratio range)
  - drop_rel = 1.33 (>30% threshold — good)
  - BUT p = 0.16 (not significant) due to n=3 workloads

INSIGHT: The cliff exists and is sharp. It's driven by compression
removing numeric tokens ("hours: 174", "budget: EUR 5608") that are
essential for the aggregation task. This is exactly the compounding-error
model prediction: q^N drops below theta when the compressor strips
task-critical tokens.

The cliff may be LESS sharp for families b and c, which depend on
structural patterns (sub-task assignments, chain links) rather than
precise numbers. This would be a valuable finding.

8. PERFORMANCE OPTIMIZATIONS (APPLIED)
--------------------------------------------------------------------------------

Three optimizations applied to cut full-run time from ~20h to ~6h:

a) OLLAMA_NUM_PARALLEL=4:
   Set on GPU server. Ollama now handles 4 concurrent requests. This
   helps H4 (reader calls) and H5 (planner calls) most. The RTX 5090
   has 32GB VRAM — easily fits phi3 (2GB) + llama3.1 (5GB) + headroom
   for 4 parallel inference streams.

b) Compression caching (code change):
   Added in-memory cache keyed by (compressor, ratio, fragment_id) to
   run_h1_h2.py and (ratio, fragment_id) to run_h5.py. Since compression
   is deterministic (temp=0), same fragment at same ratio always produces
   the same output. This avoids redundant Ollama calls across:
     - Multiple seeds (H1/H2: 5 seeds, H5: 5 seeds)
     - QA scoring vs coordination scoring (H1/H2: was compressing 3x)
     - Multiple planner models (H5: 3 models share same compressed input)
   Net effect: phi3-extractive calls drop from ~12,000 to ~1,200 for H1/H2,
   and H5 compresses once per ratio instead of once per (model, ratio).

c) Avoid model switching:
   H1/H2 iterates compressors in the outer loop (all lingua2 ratios first,
   then all phi3 ratios, then all filter ratios). This minimizes Ollama
   model load/unload cycles (~5s each). H5 pre-compresses all fragments
   before iterating over planner models.

d) Run experiments sequentially, not in parallel:
   Running multiple experiments simultaneously on the same GPU causes
   Ollama model thrashing (phi3 ↔ llama3.1 swaps, ~5s each). The
   OLLAMA_NUM_PARALLEL=4 helps within a single experiment (concurrent
   requests to the SAME model), not across experiments using DIFFERENT
   models. Full runs should chain: H1/H2 → H5 → H4 → H3.

ESTIMATED SAVINGS:
  H1/H2: ~10h baseline → ~4h (caching cuts phi3 calls 10x)
  H5: ~6h baseline → ~2h (pre-compression + parallel planner calls)
  H3: ~3h baseline → ~3h (no Ollama calls, already fast)
  H4: ~3h baseline → ~1.5h (parallel reader calls via NUM_PARALLEL)
  Total: ~22h → ~10.5h (sequential execution, no GPU contention)

12. CRITICAL BUG: QA_F1 AND TOKEN_RECALL ARE ALWAYS ZERO
--------------------------------------------------------------------------------

LOCATION: src/m6/experiments/run_h1_h2.py lines 179-189 (single_agent_qa)
          and lines 102-115 (token_recall)

PROBLEM:
  single_agent_qa() computes F1(concatenated_compressed_text, expected_answer).
  For family-a, expected_answer = "hours=819;budget=170969".
  Compressed text = "hours: 174. Approved budget: EUR 5608. ..." (raw fragments).
  Token overlap is zero because:
    a) "819" doesn't appear in any fragment (it's the SUM of individual hours)
    b) "170969" doesn't appear in any fragment (it's the SUM of budgets)
    c) The format is different ("hours=819" vs "hours: 174")

  token_recall() has the same problem — it looks for gold_answer tokens
  in compressed text, but the gold answer is aggregated.

IMPACT: H1's Spearman correlation is computed on constant-zero arrays,
  giving rho=0.0 trivially. This makes H1 "supported" for the wrong reason
  (both deltas are zero, not because QA and coordination are uncorrelated).

FIX OPTIONS:
  a) For family-a: single_agent_qa should compute the same aggregation
     (sum hours, sum budget) from the compressed text, then compare to
     expected. This mirrors what the deterministic solver does.
  b) For token_recall: measure whether individual fragment-level facts
     (e.g., "hours: 174") survive compression, not whether the aggregated
     answer does.
  c) Alternative: use the family-aware _score_answer() from run_h5.py,
     which extracts numbers with regex and scores by relative error.

PRIORITY: HIGH — H1 depends on having non-trivial qa_f1 values.

13. DETERMINISTIC SOLVER LIMITATIONS
--------------------------------------------------------------------------------

The deterministic solver (_solve_family_a, _solve_family_b, _solve_family_c)
in src/m6/agents/orchestrator.py:

STRENGTHS:
  - No LLM noise — isolates compression effects perfectly
  - Fast — regex parsing, no Ollama calls
  - Correct upper bound — uses expected_solver for subtask assignments

WEAKNESSES:
  - subtask_acc = 1.0 always (uses ground truth, not predictions)
  - coord_success is binary (1 or 0) — loses fine-grained signal
  - For family-a: regex "hours:\s*(\d+)" and "budget:\s*EUR\s*(\d+)"
    are fragile — if compression changes "hours: 174" to "hours 174"
    (drops colon), the regex still works. But if it drops "hours"
    entirely or merges tokens, the solver gets 0.

  The sharp cliff at 4x happens because LLMLingua-2 at 4x keeps enough
  tokens for the regex to fire, but drops some numbers. The solver either
  gets ALL 8 numbers right or fails. There's no partial credit.

FOR THESIS: The deterministic solver is the right choice for H1/H2
  because it isolates compression effects. But H5 (with real Ollama
  planner) shows the cliff is even sharper because LLMs are more
  sensitive to information loss than regex.

15. PHI-3 FALLBACK FIX (2026-05-24)
--------------------------------------------------------------------------------

PROBLEM:
  phi3-extractive fell back to lingua2 100% of the time in full runs,
  despite the 15% violation rate fix from commit 136305f. The verifier
  still rejected outputs where >15% of tokens were novel.

ROOT CAUSE:
  The _try_extract() method returned None on ANY verifier failure, even
  when the output was 80%+ extractive. This triggered the lingua2 fallback
  for every fragment, making phi3 results identical to lingua2.

FIX (commit 63bf8ee):
  When extractive fraction ≥ 50%, strip novel tokens and return the result
  instead of falling back. The _strip_novel_tokens() function already
  enforces 100% extractiveness on the output.

  Pipeline: model generates → verifier checks →
    if ≥85% extractive: strip & return (same as before)
    if 50-85% extractive: strip & return (NEW — salvage instead of fallback)
    if <50% extractive: retry with strict prompt → fallback to lingua2

VALIDATION:
  Tested on 20 benchmark fragments: 0/20 fallbacks (was 100% before).
  phi3 produces distinct output from lingua2 at all ratios.

RISK:
  Stripped output may lose coherence (function words removed). But the
  experiment metrics (coord_success, F1) measure content preservation,
  not readability, so this is acceptable.

17. CODE AUDIT FINDINGS (2026-05-24) — POTENTIAL ISSUES FOR FULL RUN
--------------------------------------------------------------------------------

Issues identified by code audit that may affect full production runs:

a) H1 DEGENERATE CASE HANDLING:
   When delta_coord has zero variance (e.g., all coord_success=1.0 or
   all 0.0 within a compressor group), spearmanr returns NaN → code
   forces rho=0.0 → passes H1 test. This is now MOOT because qa_f1
   provides variance, but could recur if a compressor produces constant
   qa_f1 (unlikely with the new metric).

b) H2 NO FIT-QUALITY GATE:
   Mann-Whitney runs regardless of piecewise fit quality (rmse). A high-
   rmse fit could pass the test from group variance, not a real cliff.
   Current rmse values (0.12-0.30) seem reasonable. Monitor in full run.

c) H5 ALL-NaN VACUOUS MONOTONICITY:
   If all planner models produce NaN tau for a family (e.g., family-b
   with 0% success for all models), Python's all() on empty iterator
   returns True → family counts as monotonic for free. This inflates
   the "monotonic families" count.

d) FAMILY-A SCORING THRESHOLD:
   _score_family_a uses strict inequality (f1 > 0.75). If both hours
   and budget are exactly at 25% error (f1=0.75), coord_success=0.
   This is conservative but may discard borderline successes.

e) FAMILY-B SINGLE FRAGMENT:
   Family-b workloads have only 1 fragment (~135 chars). The deterministic
   solver's regex is robust enough to parse it under compression, so
   coord_success=1.0 at all ratios. The cliff is only visible through
   qa_f1, not coord_success. This is fine for H2 (cliff detected) but
   the thesis should note that family-b's cliff is information-loss,
   not coordination-failure.

20. FAMILY-B ROOT CAUSE — AGENT vs WORKER NAMING BUG (2026-05-24)
--------------------------------------------------------------------------------

PROBLEM:
  All 3 LLM planners score 0% on family-b at ALL ratios, even ratio=1.0.

ROOT CAUSE:
  The benchmark fragment says "Agents: 5 with capacities [5, 4, 2, 4, 5]"
  so all LLMs naturally output "sub-0=agent-0; sub-1=agent-3; ..." using
  the word "agent" from the fragment text.

  But the scoring regex requires "worker-\d+":
    re.finditer(r"(sub-\d+)\s*=\s*(worker-\d+)", answer)

  Since the LLMs say "agent" not "worker", zero assignments are parsed,
  giving 0% score even when the LLM solved the problem correctly.

  Tested with all 3 models:
    qwen2.5:1.5b → sub-0=agent-3;sub-1=agent-2;sub-2=agent-4;...
    phi3:latest  → sub-0=agent-5;sub-1=agent-1;sub-2=agent-3;...
    llama3.1:8b  → (long explanation, no formatted answer)

FIX OPTIONS:
  a) Change format hint to say "agent-X" instead of "worker-X"
  b) Change scoring regex to accept both "worker-\d+" and "agent-\d+"
  c) Change benchmark fragment text to say "Workers" instead of "Agents"
  d) Accept any "sub-\d+=\w+-\d+" pattern

ADDITIONAL ISSUE:
  Even if naming is fixed, the expected_answer was generated by a FCFS
  greedy algorithm (not FFD). The LLM planner may find a DIFFERENT but
  equally valid solution. The scoring does partial matching (f1 > 0.5)
  so this is partially mitigated, but a feasibility checker would be
  better than exact assignment matching.

PRIORITY: Medium — family-b is not essential (H5 can work with family-a
  and family-c alone). But fixing it gives 3/3 families which strengthens
  the results.

21. FAMILY-B FEASIBILITY CHECKER FIX (2026-05-25)
--------------------------------------------------------------------------------

Replaced exact-assignment matching with feasibility checking for H5
family-b scoring (commit 9d006f7).

OLD: Compare LLM assignments against one specific greedy solution.
     Any different (but valid) packing scored 0%.

NEW: Check (1) all tasks assigned, (2) no agent exceeds capacity.
     Any valid bin-packing solution scores coord_success=1.0.
     f1 = fraction of tasks assigned (coverage).

RESULTS (5 workloads, 3 ratios, 1 seed):

  coord_success:        1.5B  3.8B   8B
    ratio=1.0            0.0   0.0  0.0
    ratio=4.0            0.0   0.0  0.0
    ratio=8.0            0.0   0.0  0.2

  coverage (f1):        1.5B  3.8B   8B
    ratio=1.0            1.0   1.0  0.71
    ratio=4.0            1.0   0.8  0.20
    ratio=8.0            1.0   0.8  0.53

INSIGHT: Models assign tasks (high coverage) but VIOLATE capacity
  constraints (low coord_success). The bin-packing problem is genuinely
  hard for small LLMs — they don't track cumulative load per agent.
  This is a real model limitation, not a scoring bug.

  Ironically, 1.5B has 100% coverage but 0% feasibility — it assigns
  every task to some agent but ignores capacities entirely.

  Family-b remains hard for H5 but the scoring is now correct.

24. LITERATURE GAP ANALYSIS (2026-05-25)
--------------------------------------------------------------------------------

Read all papers in papers_markdown/. Key gaps the thesis fills:

GAP 1: COMPRESSION × COORDINATION QUALITY (H1-H2)
  No paper studies the coupling between compression ratio and multi-agent
  coordination success. Compression papers measure QA F1; multi-agent papers
  assume full context. M6 is the first to measure the cliff.

GAP 2: PIPELINE PLACEMENT (H3)
  RECOMP and SARA compare compression strategies within a fixed pipeline.
  No study systematically evaluates P1 vs P2 vs P3 across regimes.
  M6's finding that compress-first (P1) always wins contradicts
  LongLLMLingua's design rationale.

GAP 3: INFERENCE DISCLOSURE VIA COMPRESSION (H4)
  Compression papers focus on task performance, not privacy side-channels.
  M6's finding that compression aggressiveness inversely correlates with
  disclosure is novel and connects quality (H2) to privacy (H4).

GAP 4: MODEL-SIZE INVARIANCE OF CLIFF (H5)
  No paper varies both model size AND compression ratio systematically.
  M6's finding that cliff position is compressor-driven (not model-driven)
  is a clean, publishable insight.

GAP 5: COORDINATION-SPECIFIC METRICS
  MultiAgentBench uses milestone KPIs but doesn't measure compression
  effects. M6's qa_f1 (information preservation) enables new evaluation.

CLOSEST RELATED WORK (that does NOT overlap):
  - PROVENCE (Chirkova 2025): compression evaluation framework, but
    single-agent only. M6 extends to multi-agent coordination.
  - MultiAgentBench (Zhu 2025): multi-agent benchmark, but no compression.
  - MaaS (Li 2025): memory as a service, but no quality measurement.
  - Yu 2026 (Multi-Agent Memory): memory architecture, no compression cliff.

POSITIONING: M6 sits at the intersection of compression × coordination ×
privacy. No existing paper occupies this intersection.
