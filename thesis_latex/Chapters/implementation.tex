This chapter presents the system design and the implementation of the
reference memory bus, the compressor framework, the C1 benchmark, the RAG
pipeline catalogue, and the agent runtime. The intent is to give a reader
enough detail to reproduce the system from the open-source repository
without needing to refer back to the code, and to make the design choices
explicit so that the experimental results in Chapter~\charef{experiments}
can be interpreted against the constraints those choices imposed. The
chapter closes with a section on the inference-backend stack, the
single-machine compute envelope, and the reproducibility package that
accompanies the manuscript.

\section{System Architecture}
\label{sec:architecture}

\subsection{Three-layer memory bus}

The reference implementation organises the memory bus into three layers
that match the access pattern of a multi-agent workflow:

\begin{enumerate}
\setlength\itemsep{2pt}
\item An \textbf{access layer}, exposing read, write, subscribe, and audit
endpoints via FastAPI, and enforcing per-request access-control checks
through a policy middleware.
\item A \textbf{compression layer}, providing a pluggable
\texttt{Compressor} protocol behind which four concrete variants live:
the identity (no-compression) control condition, the LLMLingua-2
hard-prompt baseline, the instruction-aware heuristic filter, the
ICAE-style soft-prompt compressor, and the C4 tag-preserving variant.
\item A \textbf{storage layer}, providing three protocols ---
\texttt{AuditLog}, \texttt{Scratchpad}, and \texttt{VectorStore} ---
implemented for the master's-scale evaluation by SQLite in WAL mode, an
in-memory time-to-live dictionary, and FAISS-CPU with an HNSW index.
\end{enumerate}

The three layers are summarised in
Figure~\ref{fig:architecture}. Every component dependency in the source
tree is one-directional: the compression layer does not import from the
access layer, and the storage layer does not import from either. This is
not stylistic; the goal is to allow the doctoral phase to swap the
storage layer for Postgres, Redis Cluster, and Qdrant without touching
the compression or access layers.

The three-layer structure is a direct extension of
MemIndex~\cite{saleh2025memindex}, the FCG group's baseline architecture
for distributed agent memory. The two material differences are that
compression is promoted to a first-class layer rather than treated as a
downstream optimisation, and that per-slot tags are part of the data
model rather than out-of-band metadata.

\subsection{Data flow for a write}

A single write request flows through the bus in the following order. The
client sends \texttt{POST /v1/write} with a JSON body containing the
fragment text, its tag vector, and an optional task hint. The policy
middleware parses the requester principal from the
\texttt{X-M6-Principal} header and writes it to
\texttt{request.state.principal} as a Starlette \texttt{State()}
attribute. The service receives the request, checks that the principal's
access-control mask is a superset of the fragment tag's mask and that
the principal's classification is at least the fragment's
classification, calls the active compressor's \texttt{compress(fragment,
target\_ratio)} method to produce a \texttt{CompressedSlot}, places that
slot in the in-memory scratchpad keyed on the slot identifier, computes
an embedding for the slot if the compressor exposes one and adds it to
the FAISS index, and finally appends a row to the audit log whose
\texttt{event\_type} is \texttt{WRITE}, \texttt{result} is \texttt{OK},
\texttt{prev\_hash} is the previous row's chain hash, and
\texttt{chain\_hash} is $\textsc{SHA-256}(\texttt{prev\_hash} \mathbin\Vert
\texttt{payload\_hash})$. The response carries the slot identifier, the
chain hash hex-encoded, and the audit row identifier.

If the policy check fails, the bus appends a \texttt{DENY} row with the
synthesised slot identifier \texttt{deny-<uuid8>}, with a $60$-second
per-\textit{(subject, fragment-id)} deduplication so that an adversarial
client hammering denied writes cannot inflate the audit log.

\subsection{Data flow for a read}

A read request flows symmetrically. The client sends \texttt{GET
/v1/read/\{slot\_id\}}, the policy middleware attaches the principal,
the service looks up the slot in the scratchpad, checks the
principal--tag relationship, and either returns the slot together with
the audit row identifier or raises a policy-denied or slot-not-found
error. The slot-not-found path is itself audited with a 60-second
per-\textit{(subject, slot-id)} deduplication: the first read miss for a
given subject and slot is appended to the audit log as a
\texttt{READ}/\texttt{ERROR} row, and subsequent misses within the
window short-circuit silently so that an adversarial reader cannot
inflate the audit log either.

\subsection{Boundary with the FCG platform}

The FCG software architecture~\cite{fcgsoftwarearch2026} specifies a
dual messaging topology: NATS for the real-time path with a 1--5~ms
broadcast budget, and Kafka for the durable path as the source of truth
and audit trail. The thesis collapses these into SQLite, an in-memory
dictionary, and an HTTP API for two reasons. First, the compute envelope
is one machine: running NATS and Kafka on the same M4 Pro adds
operational complexity without adding empirical signal. Second, the FCG
architecture document budgets the total synchronous epoch path at
$150$--$200$~ms end-to-end (pre-epoch $10$~ms, mechanism execution
$50$--$100$~ms, commitment $20$~ms, procurement $50$~ms, settlement
$20$~ms)~\cite{fcgsoftwarearch2026}, which SQLite and FastAPI sit
comfortably under on a single host. The contract that
FCG adopts is the FastAPI surface; whether the wire is NATS, Kafka, or
HTTP is a layer-2 decision that the doctoral phase makes.

What is preserved structurally from the FCG architecture so that
doctoral-phase substitution is straightforward: the event-type taxonomy
(\texttt{SliceRegistered}, \texttt{BidSubmitted},
\texttt{EpochResult}, \texttt{TranscriptCommit},
\texttt{AuditSignal}) is mirrored as a set of Pydantic models in the
schema module; the audit-log table schema mirrors the
\texttt{audit\_transcript} table from \cite{fcgsoftwarearch2026} with
the auction-specific columns collapsed into a single
\texttt{payload\_hash}; and the transcript hash chain
$H_t = \textsc{SHA-256}(\texttt{epoch\_id} \mathbin\Vert
\texttt{bids\_hash} \mathbin\Vert \texttt{alloc\_hash} \mathbin\Vert
\texttt{payments\_hash} \mathbin\Vert H_{t-1})$ is implemented in the
audit-log module with a tamper-detection unit test.

\section{Data Model}
\label{sec:datamodel}

\subsection{Tags, classifications, and access control}

Every fragment and every compressed slot carries a tag vector that
records its provenance and access-control state. The tag vector has
four fields: a \texttt{uint64} access-control mask, a 5-tier
classification level (\textsc{public} $<$ \textsc{internal} $<$
\textsc{confidential} $<$ \textsc{restricted} $<$
\textsc{secret}) drawn from the NIST AI Risk Management
Framework~\cite{nistairmf}, a tuple of source identifiers
recording provenance, and a tuple of parent-slot identifiers
recording which previously-compressed slots this slot inherits from.

Tag aggregation respects the lattice: when slots are merged or when a
new slot inherits from several parents, its access-control mask is the
bitwise OR of the parents' masks and its classification is the maximum.
The lattice rules are encoded in the
\texttt{TagVector.union(\ldots)} classmethod and unit-tested. The
policy predicate is symmetric for reads and writes: a requester
satisfies a tag if and only if every required bit is in the requester's
mask and the requester's classification is at least the tag's
classification.

\subsection{Compressed slot payload variants}

The compressed slot carries a tagged-union payload because the three
compressor families produce three different representations: a literal
text summary for the LLMLingua-2 and instruction-aware filter
families, a soft embedding tensor of shape $(M, d_\text{model})$ for the
ICAE family, and an integer token-identifier tuple for the
AutoCompressor-style variant. Storage and access layers never branch on
the payload kind; the embedding interface exposed by the compressor is
the only place where the kind matters.

\subsection{Audit log}

The audit log is an append-only SQLite table with the columns shown
in Table~\ref{tab:audit_columns}. An \texttt{INSTEAD OF UPDATE} and an
\texttt{INSTEAD OF DELETE} trigger refuse any mutation of an existing
row. The chain-hash trigger is verified by a dedicated unit test that
drops the trigger to simulate a file-level tamper, edits a single
\texttt{payload\_hash}, and confirms that the
\texttt{verify()} chain-walker reports failure.

\subsection{Cost ledger}

A second SQLite table records every paid model call: provider, model,
input tokens, output tokens, EUR cost, and wallclock. The cost ledger
shares the audit-log database for transactional consistency but is
queried separately. The pricing table is a single Python module
(\texttt{m6.pipelines.cost\_model}) whose constants are grounded in
the financial analysis~\cite{fcgfinancial2026}: the frontier-cloud
reference price is USD~3 per million input tokens and USD~15 per
million output tokens, which corresponds to approximately
EUR~2.76 and EUR~13.80 respectively; the on-premises ConfidentialMind
amortised marginal cost is EUR~0.05 per million tokens. The
GPT-4o-mini and Claude Haiku 4.5 prices are drawn from TalentAdore
production accounts as of the writing of this thesis.

\section{Compressor Framework}
\label{sec:compressors}

\subsection{The \texttt{Compressor} protocol}

Every compressor variant conforms to a Python \texttt{Protocol} with
four methods --- \texttt{compress(fragment, target\_ratio)} returning a
\texttt{CompressedSlot}; \texttt{decompress(slot)} returning a
best-effort text reconstruction; \texttt{embed(slot)} returning an
optional embedding for vector-store indexing; and \texttt{model\_card()}
returning a structured card containing the compressor identifier,
family, base model, training-loss formula, default target ratio, and
provenance hashes. The thesis ships five concrete implementations: the
identity (no-compression) control condition, LLMLingua-2, the
instruction-aware filter, the ICAE-style soft-prompt compressor, and the
tag-preserving variant that subclasses ICAE.

The identity compressor is the no-compression control condition
required by every experiment as a mitigation against runtime-bug
confounds. The experimental
protocol in Chapter~\charef{experiments} flags an experiment as
\texttt{invalid} whenever the variance of the control condition is
comparable to or exceeds the variance of the treatment of interest.

\subsection{LLMLingua-2 wrapper}

The LLMLingua-2 wrapper is a thin adapter around the
\texttt{llmlingua}~package~\cite{pan2024llmlingua2}. Construction takes a
target compression ratio; the package is asked to retain $1/r$ of the
tokens. The force-tokens list (sentence boundary marks) is validated
against the wrapped tokenizer's vocabulary at initialisation and unknown
tokens are dropped with an explicit log line; this avoids silent
compression-quality degradation when the wrapped model's tokenizer does
not include a punctuation mark on the default list.

Calibration target: NarrativeQA F1 within $2$~pp of the reported
numbers~\cite{kocisky2018narrativeqa} and HotpotQA F1 within $3$~pp of
the reported numbers~\cite{yang2018hotpotqa}; failure to meet these
targets triggers a stop-the-line review.

\subsection{Instruction-aware filter}

The instruction-aware filter is a pure-Python heuristic baseline that
serves as the floor against which the trained ICAE compressor must
demonstrate a benefit. It operates in two stages: first a sentence-level
filter that ranks sentences by their relevance to the task hint using
either a cross-encoder reranker
(\texttt{BAAI/bge-reranker-base}) when available or a lexical-overlap
fallback when not; second a token-level trim that drops stop-words and
short tokens until the target compression ratio is reached. The filter
is deterministic given the task hint, the fragment, and the target
ratio. It carries no training cost and is therefore the most economical
baseline; the trained ICAE compressor must beat it by at least $3$~pp
F1 to justify its training cost.

\subsection{ICAE-style soft-prompt compressor}

The trainable arm of the thesis follows the ICAE
construction~\cite{ge2024icae} with the
training objective extended to a dual-objective loss following
AutoCompressor~\cite{chevalier2023autocompressor}:
\begin{equation}
L = L_\text{recon} + \lambda \cdot L_\text{NCE}
\label{eq:dual_objective}
\end{equation}

The architecture (Section~\ref{sec:training}) uses
Llama-3.1-8B-Instruct~\cite{llama31modelcard} as both encoder and
decoder. The encoder is LoRA-adapted~\cite{hu2022lora} with rank $16$,
alpha $32$, and dropout $0.05$ on the seven canonical target modules
(\texttt{q\_proj}, \texttt{k\_proj}, \texttt{v\_proj},
\texttt{o\_proj}, \texttt{gate\_proj}, \texttt{up\_proj},
\texttt{down\_proj}). The decoder is frozen. The memory consists of
$M = 128$ learnable special tokens appended to the encoder input;
their last-layer hidden states are the compressed representation. The
reconstruction loss $L_\text{recon}$ is the standard token-level
cross-entropy of the source fragment given the compressed memory slots,
computed on the frozen decoder.

The contrastive loss $L_\text{NCE}$ is an InfoNCE
loss~\cite{oord2018cpc} over positive and negative source-fragment
pairs. Positive pairs are drawn from fragments of the same source
document or from fragments observed during the same
planner-worker-critic dialogue trace (the latter is the H3 dialogue
variant). Negative pairs are other fragments in the same batch
(in-batch negatives), and the temperature is $0.07$ following
SimCLR~\cite{chen2020simclr}. The weight $\lambda$ is searched in
$\{0.1, 0.3, 1.0\}$ on a validation split.

The InfoNCE primitive in MLX is not available as of version $0.21$, so
the trainer probes for it at startup and falls back to a PyTorch-MPS
implementation if the probe fails; this is the documented mitigation
for the second risk-register row of the implementation plan. The
\texttt{infonce\_loss} function is unit-tested against a numpy reference
implementation to confirm numerical equivalence.

\subsection{Tag-preserving variant (C4)}
\label{sec:c4_design}

The C4 variant is a subclass of the ICAE compressor that adds a per-slot
tag-prediction head. The tag head is a two-output MLP that emits, from
each pooled memory-slot embedding, a 64-dimensional logit vector for the
access-control mask and a 5-dimensional logit vector for the
classification level. The training loss extends
Equation~\ref{eq:dual_objective} with a tag-prediction term:
\begin{equation}
L = L_\text{recon} + \lambda_\text{NCE} \cdot L_\text{NCE}
       + \lambda_\text{tag} \cdot L_\text{tag},
\label{eq:c4_loss}
\end{equation}

where $L_\text{tag}$ is binary cross-entropy on the ACL output and
cross-entropy on the classification output. The weight
$\lambda_\text{tag}$ is searched in $\{0.1, 0.3, 1.0, 3.0\}$. Tag
recovery at inference time is performed slot by slot: ACL bits are
recovered via a 0.5 sigmoid threshold and aggregated across slots by
bitwise OR; the classification is recovered as the argmax of the
softmax and aggregated by taking the maximum across slots. The
aggregation is lattice-respecting by construction.

The H5 evaluation defines a fragment's tag as \emph{preserved} when
either the recovered ACL is a superset of the source ACL or
$|\text{recovered} \cap \text{source}| / |\text{source}| \geq 0.9$,
and the recovered classification is at least the source classification.
The threshold of $0.9$ is the H5 falsification target.

\subsection{Training configuration}
\label{sec:training}

Optimiser AdamW with learning rate $3 \times 10^{-4}$, weight decay
$0$, $\beta_1 = 0.9$, $\beta_2 = 0.95$, cosine schedule with 200-step
linear warmup. Batch size 4, gradient accumulation 8, sequence length
512 (fragment) plus 128 (memory). Wallclock budget at most three days
on the M4 Pro 48~GB for a 7B-parameter encoder at rank 16. A
13B-parameter encoder at rank 8 is the upper limit and is treated as a
stretch goal; the H7 (model-size scaling) results rely on
inference-only scaling beyond 7B if the 13B fine-tune does not fit in
memory.

Training data for the H3 distribution-matters comparison consists of
two datasets of approximately 5\,000 examples each: a question-answering
distribution drawn from public NarrativeQA and HotpotQA-style data, and
a multi-agent dialogue distribution synthesised by running the C1
generator with the identity (no-compression) compressor and capturing
clean planner-worker-critic traces. The two datasets are
training-budget-matched so that the H3 comparison isolates the
distribution variable.

\section{The C1 Benchmark}
\label{sec:c1}

\subsection{Workload families}

The C1 benchmark generator produces three workload families:

\begin{description}
\item[Family (a) --- cross-document fact aggregation.] Eight source
documents per instance, each representing one institutional system in
the Vignette~3.7 inventory (Publication~DB, Contract~DB, Moodle, Patio,
Peppi, CRM, Tatu~SAP, SAP~Travel~\cite{fcgusecase2026}); the planner is
asked to aggregate hours and budget across the
eight sources. The aggregation is deterministic by construction so that
the ground truth is reproducible from the seed; the critic checks the
planner's answer against the aggregate.

\item[Family (b) --- constraint-satisfaction planning.] $N$ sub-tasks
must be assigned to $K$ capacity-bounded workers under
load-cum-capacity constraints; the workload generator produces a
feasible solution via a greedy capacity-respecting algorithm that bumps
capacities only when strictly necessary and retries the same sub-task,
guaranteeing that every workload has at least one feasible assignment.

\item[Family (c) --- multi-step retrieval.] A linear chain of fragments
where each fragment points to the next and the leaf carries the answer;
chain lengths range from $2$ to $4$ and each fragment is padded with
$4$ to $8$ distractor sentences so that the retrieval step is non-
trivial.
\end{description}

Each family produces approximately $50$ instances per generation seed
for a total of $150$ instances per release. The release is reproducible
from a single command and the configuration hash is recorded in the
manifest for forensic comparability.

\subsection{Synthetic tag distributions}

The thesis uses synthetic tags because real ACL data was not
available on the master's-scale schedule. The generator supports three
tag distributions: \emph{uniform}, in which each ACL bit is set
independently with probability $0.5$; \emph{skewed}, in which a handful
of bits dominate via an exponential family chosen to mimic typical
production ACL skews; and \emph{hierarchical}, in which higher
classification levels imply strict supersets of the bits at lower
classification levels. The H5 and H6 evaluations vary the distribution
to probe the governance-stringency dimension explicitly.

\subsection{Coordination-quality metrics}

The coordination scorer is a pure function of the workload and the
trace; it does not call the model. It returns four numbers:
\emph{final task success} (boolean, family-specific predicate),
\emph{sub-task assignment accuracy} (the fraction of sub-tasks
assigned to the worker the workload's ground truth expects),
\emph{rounds to completion} (the number of planner-worker-critic
cycles before the planner declares \texttt{DONE}), and
\emph{critic-flagged error rate} (the fraction of rounds in which the
critic returned \texttt{REVISE}). The summary-level inference disclosure
proxy is computed by a separate path in
Section~\ref{sec:h6_design} of Chapter~\charef{experiments} and
involves a held-out reader model.

The success predicate for family (a) is an exact match on the
aggregate, for family (b) is a constraint-satisfaction check against
the workload's advertised capacities, and for family (c) is an exact
match on the leaf answer. The constraint-satisfaction predicate fails
closed when the workload's metadata is missing the capacities field,
which would otherwise be an opportunity for the planner to pass
trivially under a generator bug.

\section{RAG Pipeline Catalogue}
\label{sec:rag_pipelines}

The C3 catalogue consists of three pipelines, each implemented on top
of FAISS-CPU and LlamaIndex~\cite{faiss, llamaindex}.

\subsection{P1: compress, then retrieve}

The corpus is compressed up-front; the FAISS index stores the
compressed text or, for soft-prompt compressors, the mean-pooled
memory-slot embeddings. Retrieval and synthesis operate on the
compressed representation. P1 trades quality for storage efficiency:
information dropped at compression time is gone, but the index is small.
The pre-processing baseline against which P1 is informally compared is
Anthropic's contextual-retrieval~\cite{anthropic2024contextualretrieval},
which prepends an LLM-generated chunk-level context summary before
embedding.

\subsection{P2: retrieve, then compress}

The corpus is indexed in full; the compressor runs as a post-retrieval
node-processor over the top-$k$ retrieved fragments before they are
synthesised. This is the classical LongLLMLingua
configuration~\cite{jiang2024longllmlingua}. P2 trades storage cost for
retrieval recall: the index is the full corpus, but the synthesis
context can be compressed aggressively because the retriever has already
identified the relevant chunks.

\subsection{P3: joint, conditional on relevance}

The corpus is indexed in full. At query time, each retrieved fragment is
routed into one of three branches based on its retrieval relevance score
$s$: fragments with $s \geq \theta_\text{high}$ are passed verbatim,
fragments with $\theta_\text{low} \leq s < \theta_\text{high}$ are
compressed, and fragments with $s < \theta_\text{low}$ are discarded.
The thresholds $\theta_\text{high}$ and $\theta_\text{low}$ are
swept over a small grid; the default values $0.75$ and $0.45$ on
\texttt{BAAI/bge-large-en-v1.5}-cosine similarities are the analytical
starting points. P3 follows the structural design of
\cite{guo2025dynamic}.

\subsection{Cost model}

The pipeline cost model is a single Python module whose pricing
constants are grounded in the financial-analysis
document~\cite{fcgfinancial2026}: the frontier-cloud reference price is
USD 3 per million input tokens and USD 15 per million output tokens, or
approximately EUR 2.76 and EUR 13.80. The on-premises amortised
marginal cost is EUR 0.05 per million tokens. The
GPT-4o-mini and Claude Haiku 4.5 prices reflect TalentAdore production
accounts as of the writing of this thesis. The headline cost metric in
the C3 evaluation is EUR per workflow, with a corollary headline
of EUR per period-end report against the
Vignette~3.7 target of approximately EUR~2.15~\cite{fcgusecase2026}.

\section{Agent Runtime}
\label{sec:agents}

The agent runtime uses AutoGen v0.4 pinned in the
\texttt{requirements.txt}~\cite{wu2023autogen}. The planner-worker-
critic loop is built as a \texttt{RoundRobinGroupChat} over a planner,
a configurable number of workers, and a critic that emits a structured
\texttt{DONE} or \texttt{REVISE} verdict. The scratchpad is the memory
bus: every agent write goes through \texttt{POST /v1/write} and every
agent read goes through \texttt{GET /v1/read/\{slot\_id\}}, so the
compressor is on the critical path and the experimental knob is what
the experiment manipulates rather than a hidden detail of the runtime.

A second backend, \texttt{determ}, is a deterministic family-specific
solver that reads the (optionally compressed) fragments and produces a
trace directly. Its purpose is not to replace AutoGen as the production
runtime; it is the \emph{upper bound} on coordination success that
isolates compression-quality variance from runtime variance. If the
AutoGen runner fails to match the deterministic runner on the
no-compression control condition, the experiment is flagged invalid in
the manifest.

The orchestrator caches the constructed compressor by
$(\text{name}, \text{ratio})$ so that a trained ICAE compressor is
instantiated once per cell rather than once per workload--seed pair;
without this cache the model would re-load on every inner-loop
iteration of H2 (approximately twenty-two thousand iterations per
run).

\section{Inference Backends}
\label{sec:inference}

The unified \texttt{InferenceBackend} protocol exposes a single
\texttt{complete(prompt, max\_tokens, temperature, stop)} method,
together with an optional \texttt{embed(text)} method and a cost
predicate that consults the pricing table. Five concrete backends ship:
MLX-LM for $7$B and $13$B inference on Apple
Silicon~\cite{mlxframework}; llama.cpp with GGUF Q4\_K\_M for $34$B
and $70$B inference~\cite{llamacpp}; Ollama wrapping a localhost HTTP
endpoint for developer-iteration speed~\cite{ollama}; OpenAI for
GPT-4o-mini as the H6 judge and as the cost-arm comparator; and
Anthropic for Claude Haiku 4.5 as the cost-arm comparator and Claude
Sonnet 4.6 as the cross-check on the 10\% sample. A sixth backend,
\texttt{echo}, is a deterministic mock used in unit tests.

The OpenAI backend dispatches between \texttt{max\_tokens}
(GPT-4o-mini and other non-o-series models) and
\texttt{max\_completion\_tokens} (o-series) on the model name; passing
the wrong parameter is rejected by the API in some SDK versions.

The llama.cpp backend runs a five-prompt NarrativeQA-shaped sanity
check at construction and refuses to serve traffic if F1 drops by more
than $5$~pp from a recorded baseline; the baseline lives in
\texttt{results/sanity/llamacpp\_baseline.json}, must be populated by a
Month-1 run, and the backend refuses to start if it is missing. This
realises the kill-switch documented in the implementation plan's risk
register as the mitigation for the 70B int4-output-quality risk.
